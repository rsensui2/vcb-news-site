タイトル: Claude Codeを「なんとなく使ってる」から卒業する。Best Practiceを全部まとめた

Claude Code、使ってますか？

僕の周りでも使っている人は増えてきましたが、まだまだ活用しきれている人は少ないように思います。

かくいう僕も最初はそうでした。とりあえず指示を出して、返ってきたコードをコピペして、うまくいかなかったらまた聞いて…の繰り返し。正直、ChatGPTとの違いがよくわかりませんでした。

でも、GitHub（ソースコードの共有サイト）で公開されている「claude-code-best-practice」というリポジトリ（プロジェクトの保管場所）を見つけてから、使い方がガラッと変わりました。スター数は3万超え。海外の開発者がClaude Codeのベストプラクティスをまとめてくれています。（元はClaude Codeの開発者boris chemyの使い方をまとめたもの）

この記事では、そのリポジトリに掲載されている全87個のTipsを「Claude Codeをこれから使う人・使い始めたばかりの人」向けに、僕なりに噛み砕いて紹介していきます。

ちなみに、このベストプラクティス集は自分のClaude Codeに簡単に取り込めます。一番簡単なのは、Claude Codeを開いてこう打つだけです。

Claude Codeがリポジトリの内容を読み取って、プロジェクトに合った形でCLAUDE.md（後述する「記憶ファイル」）を作ってくれます。コマンドを覚える必要もありません。

ただ、以下の内容も併せて理解しておくと格段に使い方がレベルアップすると思うので、ぜひ最後まで読んでみてください。

まず知っておきたい : Claude Codeの「拡張」の仕組み

Tipsの解説に入る前に、「Claude Codeにはこういう機能がある」という全体像だけ押さえておきます。これを知らないとTipsの意味がわからないので、ここだけお付き合いください。

Claude Codeは、ターミナル上で動くAIコーディングアシスタントです。ChatGPTのような画面ではなく、エンジニアが使う黒い画面の中で動きます。普通のAIチャットとの違いは、あなたのPC上のファイルを直接読み書きできること。コードを提案するだけでなく、ファイルの作成・編集・テスト実行まで全部やってくれます。

そして、Claude Codeには以下の6つの拡張機能があります。

CLAUDE.md（メモリ）は、プロジェクトのルールや規約を書いておくファイルです。Claude Codeはセッション（会話）開始時にこれを読み込んでくれるので、毎回「うちのプロジェクトは…」と説明し直す必要がなくなります。いわば「AIへの申し送り書」です。

Commands（コマンド）は、よく使う指示をテンプレート化する仕組みです。 .claude/commands/review.md というファイルを作っておけば、 /review と打つだけでレビューが走ります。

Subagents（サブエージェント）は、メインのClaude以外に「専門AI」を立ち上げる仕組みです。「設計担当」「テスト担当」「レビュー担当」をそれぞれ独立した環境で動かせます。

Skills（スキル）は、再利用できる知識パックです。Commandsが「やることリスト」なら、Skillsは「教科書」。Claude が状況に応じて勝手に参照してくれます。

Hooks（フック）は、特定のイベント（ファイル変更やコミットなど）が起きたとき裏側で走る自動処理です。

Plugins（プラグイン）は、上記をまとめてパッケージ化し、チームやコミュニティで共有する仕組みです。

全部いっぺんに使う必要はありません。この記事のTipsを読みながら「お、これ使えそう」と思ったものから試していけばOKです。

  1. プロンプトの書き方 : 「何を言うか」で出力が変わる

Claude Codeの出力品質は、プロンプト（指示文）の書き方に大きく左右されます。リポジトリでは3つのプロンプトTipsが紹介されていますが、どれもすぐ使えるものばかりです。

Claudeに「自己チェック」させる

一番おすすめなのが「変更内容をClaudeにテストさせてからPRを出す」パターンです。

この変更をグリルして（厳しくチェックして）。

テストに合格するまでPRは作らないで。

Claude に「自分が書いたコードを自分で検証させる」わけです。人間で言えば「提出前に自分で見直して」と頼むようなもの。これだけで、ケアレスミスがかなり減ります。

「仕切り直し」のプロンプト

修正を何回か繰り返してもイマイチなとき、こう言います。

ここまでの全情報を踏まえて、今のやり方を捨てて、

エレガントな方法で実装し直して。

中途半端なパッチ（応急修正）を重ねるより、一度リセットしたほうが良い結果になることが多いです。「なんか微妙だな」と感じたら、遠慮せず仕切り直してもらいましょう。

マイクロマネジメントしない

バグを報告するときは「エラー内容を貼って『直して』と言うだけ」でOKです。

「まずこのファイルを開いて、23行目を見て、ここをこう変えて…」みたいに手順を細かく指示する必要はありません。Claudeが自分でコードを読んで、原因を特定して、修正してくれます。手順を指示すると、むしろClaudeの判断力を制限してしまうことがあるんですよね。

  1. 計画の立て方 : 「いきなり書かせない」が鉄則

リポジトリの計画系Tipsは6つあって、ここが一番「知ってるかどうかで差がつく」ポイントだと思います。

プランモードを使う

Claude Codeには「プランモード」があります。 shift + tab で切り替えると、Claudeはコードを書かずに「計画だけ」を出力してくれます。

いきなりコードを書かせると見当違いの方向に走ることがあるので、まずここで方針を確認するのが基本です。リポジトリでも繰り返し「まずプランモードから始めろ」と言われています。

Claudeにインタビューさせる

ざっくりした要件しかないとき、こう頼みます。

以下の要件を読んで、わからないところを質問して。

全部答えたら仕様書としてまとめて。

要件: ユーザーがログインできるようにしたい

Claudeが「認証方式はどれにしますか？」「パスワードリセットは必要ですか？」と聞いてきます。それに答えていくと、自然と仕様が固まっていきます。自分の頭の中にあるイメージを引き出してもらう感覚ですね。

フェーズに分ける

大きな機能を一気に作らせるのではなく、段階を区切ります。

以下のフェーズに分けて実装して。各フェーズ完了時にテストを実行すること。

フェーズ1: データモデルの定義

フェーズ2: APIエンドポイントの実装

フェーズ3: フロントエンドのUI作成

各フェーズにテストを設定しておけば、問題があった場合に早い段階で気づけます。一気に全部作らせて最後にまとめてテスト、だと手戻りが大きくなりがちです。

もう1人のClaudeにレビューさせる

サブエージェントを使って、経験豊富なエンジニアの視点で計画に穴がないかチェックしてもらう方法です。1人では気づかない問題が見つかることがあります。

仕様書は詳しく書く

リポジトリでは「仕様書を詳しく書くほど、Claudeのお守りが減る」と言われています。

「ログイン機能を作って」より「メールアドレスとパスワードでログインする機能。パスワードは8文字以上。ログイン失敗時はエラーメッセージを表示。3回失敗でロック」のほうが、出力の精度が格段に上がります。

/simplify で整理してから次へ

フェーズが終わるたびに /simplify コマンドでコードをリファクタリング（整理整頓）しておくと、後半のフェーズで問題が起きにくくなります。

  1. CLAUDE.mdの育て方 : AIを「自分仕様」にする

CLAUDE.mdに関するTipsは6つ。ここは「Claude Codeを長く使う」ほど効いてくるポイントです。

何を書けばいいか

プロジェクトの概要、チームのルール、コーディング規約（コードの書き方のルール）、設計方針の決定事項です。最初から完璧に書く必要はありません。使いながら「あ、これも書いておこう」と思ったタイミングで追加していけばOKです。

たとえばこんな感じです。

エンジニアでない方は、「このプロジェクトはPythonで書いています。ファイル名は小文字で統一してください」くらいのシンプルな内容でも十分効果があります。

毎日更新する

セッション終了時にCLAUDE.mdを更新する習慣をつけると、翌日のClaudeが「昨日の続き」をスムーズに引き継いでくれます。「今日やったこと」「次回の注意点」を数行メモするだけで十分です。

@記法で外部ファイルを読み込む

設計書やAPI仕様書（APIの使い方を定義したドキュメント）が別ファイルにある場合、CLAUDE.mdに @docs/api-spec.md と書いておけば、Claudeがそのファイルも一緒に読んでくれます。

ルールをファイルで分割する

.claude/rules/ というフォルダにルールファイルを分けて置けます。「TypeScriptのルール」「テストの書き方」「Gitの運用ルール」を別々に管理できるので、プロジェクトが大きくなっても破綻しません。

auto-memoryを使う

Claudeがメモリを勝手に更新してくれるモードもあります。「毎回更新するのは面倒…」という方は、まずはこれを有効にしてみてください。

lessons learnedを記録する

Claudeが過去に犯したミスのパターンをCLAUDE.mdに書いておくと、同じミスの再発を防げます。「このプロジェクトでは○○の書き方をすると△△のエラーが出る。□□の方法を使うこと」のような形式です。

  1. デバッグのコツ : バグ修正をうまく任せる

デバッグ（バグ修正）系のTipsは8つ。Claude Codeに修正を任せるコツが詰まっています。

情報は全部渡す

エラーのトレース（エラーの発生経路を示す情報）やログは省略せずに全部渡してください。情報が多いほど、Claudeは正確に原因を特定してくれます。

まず再現、それから修正

いきなり「直して」ではなく、「まずこのバグを再現して、それから修正して」と頼むのが鉄則です。再現できないバグは直しようがないので、ここを飛ばすと的外れな修正になりがちです。

根本原因を聞いてから直す

このバグの根本原因を説明してから、修正案を出して。

原因を理解しないまま修正に飛びつくと、別の場所で同じバグが再発することがあります。Claudeに原因分析させることで、表面的な修正ではなく根本的な解決ができます。

失敗したら巻き戻す

修正がうまくいかなかったら、チェックポイント機能で巻き戻せます。 Esc キーを2回押すか、 /rewind コマンドで、変更前の状態に戻せます。「やり直しがきく」と思うと、安心して色々試せますよね。

Webアプリは /chrome でデバッグ

ブラウザを起動して、画面の表示を確認しながらデバッグを進められます。「ボタンを押しても反応しない」みたいなUI（画面表示）のバグには特に有効です。

エッジケースを確認させる

修正が終わったら、「想定外の入力パターンでもテストして」と頼みましょう。Claudeが見落としがちな境界値（0、空文字、巨大な値など）をカバーできます。

  1. Commandsの活用 : 繰り返しをなくす

Commands（コマンド）に関するTipsは8つ。「同じ指示を何度も打っている」人ほど効果が大きいです。

よく使う指示をファイルにする

.claude/commands/review.md にレビュー用のプロンプトを書いておけば、 /review で一発実行。毎回「コードレビューして。以下の観点で…」と打つ必要がなくなりますし、レビューの観点がブレません。

役割別にコマンドを作る

フロントエンド開発用、バックエンド開発用、ドキュメント作成用、とコマンドセットを分けておくと、作業の切り替えがスムーズになります。

コマンドを連鎖させる

「まずレビュー → 問題なければテスト → 通ったらPR作成」のようにチェーン（連鎖）で使うパターンもあります。

日常チェックをコマンド化する

/daily-check のようなコマンドを作って、毎朝プロジェクトの基本動作を確認する習慣をつけると、問題の早期発見につながります。

コマンドにサンプルを入れる

コマンドファイルの中に「こういう形式で出力して」という例を入れておくと、出力の品質が安定します。

  1. Subagentsの使い方 : チームで働かせる

サブエージェントに関するTipsは6つ。「1人のClaudeでは手に負えない」規模のタスクで威力を発揮します。

役割を明確にする

.claude/agents/reviewer.md のような定義ファイルに、役割・使えるツール・使えないツールを書いておきます。レビュー担当にはファイルの読み取りだけ許可して、勝手にコードを書き換えないようにする、といった制御ができます。

独立したコンテキストを与える

リポジトリでは「エージェントには新しい独立したコンテキスト（会話の文脈）を与えて、コンテキスト汚染を防ぐ」と強調されています。1つのエージェントに色々やらせると、情報が混ざって精度が落ちるんですよね。

Agent Teamsで並列に走らせる

複数のサブエージェントを同時に立ち上げて、独立したブランチ（枝分かれした作業スペース）で並列に作業させられます。「3つの機能を実装して」→ 3エージェントが同時作業 → 完了後にマージ（統合）、という流れです。

独立して動作確認してから組み合わせる

各エージェントを単体でテストしてから、ワークフロー全体に組み込むのがおすすめです。いきなり複雑な連携を組むと、どこで問題が起きているかわかりにくくなります。

  1. Skillsの活かし方 : 知識を再利用する

Skills（スキル）に関するTipsは7つ。「何度も同じ説明をしている」と感じたら、スキルにまとめるタイミングです。

Commandsとの使い分け

Commandsは「自分で /名前 で呼び出す」もの。Skillsは「Claudeが状況に応じて勝手に参照する」もの。 .claude/skills/react/SKILL.md にReact（Webアプリを作るためのライブラリ）の規約を書いておけば、Reactのコードを書くときにさっと読み込んでくれます。

領域ごとに作る

React用、テスト用、API設計用…のようにドメイン（領域）ごとにスキルを分けると管理しやすいです。

段階的に情報を開示する

スキルの中身を最初から全部読み込ませると、コンテキストウィンドウ（Claudeが一度に扱える情報量の上限）を消費します。「必要になったときに必要な部分だけ読み込む」書き方にすると、長時間の作業でもClaudeの精度が落ちにくくなります。長時間の作業で「Claudeの応答がおかしくなってきた…」と感じるのは、大体このコンテキストが溢れているのが原因だったりします。

  1. ワークフローの組み方 : 全部をつなげる

ワークフロー系のTipsは22個（基本10 + 上級12）と最多。ここまで紹介したCommands、Subagents、Skillsを組み合わせるパターンです。

Command → Agent → Skill パターン

リポジトリが推奨する基本パターンです。 /implement と打つだけで、architect が設計 → implementer が実装 → reviewer がレビュー、まで一気に回ります。

「Research → Plan → Execute → Review → Ship」の流れ

リポジトリに掲載されている開発ワークフローの多くが、この5ステップに従っています。

調査（Research）で今のコードの状態を把握。計画（Plan）でプランモードを使って方針を立てる。実行（Execute）で計画に沿ってコードを書く。レビュー（Review）でサブエージェントにチェックさせる。リリース（Ship）でテストを通してPR（Pull Request。コードの変更をチームに提出する仕組み）を作成。

Git Worktreeで並列作業

Git Worktree（同じプロジェクトの別ブランチを同時に作業できるGitの機能）を使うと、複数のエージェントが独立した環境で同時に作業できます。お互いの変更がぶつかる心配がありません。

/schedule でクラウド実行

PCの電源が入っていなくても、クラウド上（Anthropicのサーバー）にタスクを登録して定期実行させられます。毎朝のコード品質チェックや、定期的なドキュメント更新などに使えます。

/loop でローカル連続実行

自分のPC上で最大3日間の連続実行ができます。長時間かかるタスクを放置しておくのに使えます。

  1. Git / PRの運用 : チーム開発で差がつく

Git（バージョン管理ツール。コードの変更履歴を記録する仕組み）とPR（変更内容をチームに共有して確認してもらう仕組み）に関するTipsは9つです。

PRは小さく保つ

1つのPRに複数の変更を詰め込むと、レビューが大変になりますし、問題が起きたときにどの変更が原因かわかりにくくなります。「1つのPRには1つの変更」が基本です。

コミットメッセージはClaudeに書かせる

コミットメッセージ（変更の記録に添えるメモ）は、Claudeに書かせると楽です。変更内容を理解したうえで適切なメッセージを生成してくれます。

マージ前にClaudeにチェックさせる

「mainブランチ（メインの開発ライン）との差分を確認して、互換性を壊す変更がないかチェックして」と頼むと安心です。

PRのレビューもAIにやらせる

GitHub App（GitHubの拡張機能）を使うと、PRが作られたタイミングでClaude Codeが自動でレビューしてくれます。バグ、セキュリティ上の問題、パフォーマンス（処理速度）の懸念をチェックしてくれるので、人間のレビュアーの負担が減ります。

  1. 毎日の運用 : 習慣にすると効くこと

日常の運用Tipsは9つ。地味だけど、続けるとジワジワ効いてくるものが多いです。

フォルダ構成を統一する

Claudeはプロジェクトの構造を読み取ってくれるので、一貫した構成にしておくほど精度が上がります。

Claudeの出力は必ず目視確認する

「確認なしで統合はNG」とリポジトリでも明記されています。AIは間違えることがあるので、最後のチェックは人間が担います。

.claude/rules/ をチームで共有する

チーム全員が同じルールでClaude Codeを使えるようになるので、コードの品質が揃います。

Auto Modeで許可確認を省略する

起動時に --permission-mode auto をつけると、ファイル書き込みやコマンド実行のたびに「許可していいですか？」と聞かれなくなります。作業の流れが中断されなくなるので、体感の生産性がかなり変わります。

音声入力を使う

/voice コマンドで音声入力モードに切り替えられます。キーボードを打つのが面倒なとき、考えをそのまま声で伝えてClaudeに整理してもらう使い方ができます。日本語にも対応しています。

Remote Controlでどこからでも操作

/remote-control を使うと、PCで始めたセッションをスマホやタブレットから操作できます。外出先でビルド（コードを実行ファイルに変換する処理）の進捗確認や追加指示ができて便利です。

  1. 有名な開発ワークフローの紹介

リポジトリでは、GitHub上で人気のある開発ワークフロー（Claude Codeの使い方をパッケージ化したもの）も紹介されています。自分に合ったスタイルを見つける参考になります。

Superpowers（スター数12.2万）は、テスト駆動開発（テストを先に書いてからコードを書く手法）を最重視するスタイルです。「まずテストを書いて、そのテストが通るようにコードを書く」というサイクルを徹底します。

https://github.com/obra/superpowers

Spec Kit（スター数8.3万）は、仕様書駆動のアプローチです。コードを書く前に詳細な仕様書を作り、Claudeはその仕様書に忠実に従って実装します。「仕様書の精度＝出力の精度」という考え方です。

https://github.com/github/spec-kit

Get Shit Done（スター数4.4万）は、実用重視のスタイルです。タスクごとに新しいコンテキスト（会話の枠）を使い、複数タスクを並列で一気に片付けます。

https://github.com/gsd-build/get-shit-done

どのワークフローも「調査 → 計画 → 実行 → レビュー → リリース」の基本パターンは共通しています。違いはどのステップに重点を置くか、というスタイルの差です。

まとめ : まずはCLAUDE.mdから始めよう

87個のTipsを一気に紹介してきましたが、「全部やらなきゃ」とは思わなくて大丈夫です。僕のおすすめは段階的に試していくことです。

まずはCLAUDE.mdを作ります。自分のプロジェクトのルールを10行くらいでいいので書いてみてください。これだけで出力精度が目に見えて変わります。

次にプロンプトの書き方を意識してみます。「プランモードから始める」「マイクロマネジメントしない」、これだけで体験がだいぶ変わります。

慣れてきたらCommandsを1〜2個作って、SubagentsやSkillsに手を出してみてください。

Claude Codeは「使い込むほど、自分のプロジェクトに最適化されていくツール」です。CLAUDE.mdにルールを追加するたびに、Commandsを増やすたびに、少しずつ「自分だけのAI開発環境」が出来上がっていきます。その過程がけっこう楽しいので、ぜひ試してみてください。
