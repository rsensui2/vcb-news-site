私の過去投稿⬇︎

私の自宅には、一台の古びたMacBook Pro M1が置いてある。メモリ8GB。2020年発売のマシン。今の基準で言えば、決してハイスペックではない。

そのマシンが今、13体のAIエージェントを24時間動かし続けている。

最近はMacminiを買えとうるさい発信者が多いが実際は

使ってないPCなんならアンドロイドスマホでも動かせる

Discordに指示を書き込めば、エージェントたちが議論を始め、コードを書き、成果物を納品する。私が寝ている間も、外出している間も、彼らは動いている。クラウドの高額なサーバーは使っていない。AWSもGCPも契約していない。自宅の片隅に置かれた、閉じたまま充電器につながれたノートPCが、すべての基盤。

これは、個人が自前のAIインフラを構築し、複数のエージェントを組織として運用するまでの記録。技術的な解説であると同時に、「こんなことが本当に個人でできるのか」という問いに対する、実践からの回答でもある。

大企業が何億円もかけて構築するようなAIエージェントシステムを、個人が、自宅で、ほぼ無料で作った話。

第一章 なぜ「自前」にこだわったのか

AIエージェントという言葉が急速に広まっている。ChatGPTにプラグインをつけて自動化する。AutoGPTを試す。Difyでワークフローを組む。手段はいくらでもある。

しかし、それらのほとんどはSaaS。誰かのサーバーの上で動く、誰かが決めたルールの中での自動化。便利ではある。だが、自分のものではない。

私が求めたのは「所有」だった。

自分の手元にあるマシンで動く。自分がコードを書いて制御する。データは自分のディスクに残る。APIキーの管理も、モデルの選択も、すべて自分の裁量。

SaaSが明日値上げしたら。サービスが終了したら。利用規約が変わったら。自前で持っていれば、そのリスクはゼロになる。

もうひとつの動機は、単純な好奇心。「AIエージェントを組織として動かす」とはどういう体験なのか。一体ではなく、複数。しかも、それぞれに役割を持たせ、連携させる。その可能性を、自分の手で確かめたかった。

そこで出会ったのが、OpenClawというオープンソースのフレームワーク。

第二章 OpenClawとの出会い

OpenClawは、AIエージェントの管理システム。オープンソースで公開されており、誰でも無料で使える。

特徴をひとことで言えば、「AIエージェントにインフラを与えるフレームワーク」。

単にLLMのAPIを叩くわけではない。

エージェントの起動、停止、監視。複数エージェントの並列実行。

外部サービスとの接続。記憶の永続化。定期的なタスク実行。エージェントが動作するために必要な基盤を、まるごと提供する仕組み。

私がOpenClawを選んだ理由は三つ。

一つ目は、ローカルマシンで動くこと。 クラウドインスタンスが不要。手元のMacBookにインストールして、そのまま運用できる。

二つ目は、Discordとの統合が自然なこと。 Discord上のチャンネルを指揮所として使い、メッセージを送るだけでエージェントに指示を出せる。エンジニアにとってDiscordは日常のツール。新しいUIを覚える必要がない。

三つ目は、拡張性。 自分でボットを書き、オーケストレーターを組み、エージェント同士の連携ロジックを自由に設計できる。フレームワークに縛られるのではなく、フレームワークを使いこなす感覚。

インストール自体は驚くほど簡単だった。GitHubからクローンして、依存関係をインストールして、設定ファイルにAPIキーを書く。ものの30分で最初のエージェントが動き始めた。

ただし、そこから大規模な運用にまで育てるには、設計と試行錯誤が必要だった。

第三章 システム構成の全体像

ホストマシン： MacBook Pro M1。メモリ8GB。自宅のデスクの端に常時起動の状態で置いてある。蓋は閉じたまま。ディスプレイは使わない。純粋にサーバーとして機能している。

このマシンの上でOpenClawが動き、その配下にClaw-Empireと名付けたエージェント群がポート9999で起動している。13体のBoltエージェントが並走し、APIでタスクを投入すると並列に処理を実行する構成。

指揮所はDiscord。 三つのチャンネルで運用を回している。

「指示」チャンネル → 私からの命令を受け付ける窓口

「ai議論」チャンネル → 開発タスクの議論場

「納品物」チャンネル → 完成した成果物の受け渡し場所

ネットワーク接続にはCloudflareトンネルを使っている。 通常、自宅サーバーを公開するにはドメイン取得、SSL証明書の設定、ポート開放、ファイアウォールの調整と、面倒な手順が必要になる。Cloudflareトンネルはそのすべてを省略できる。無料プランで十分。設定は数分で完了する。

そして、私が普段使うメインマシンはMacBook Pro M4 Max。メモリ48GB。こちらは外出先から持ち歩き、Discordを通じて自宅のM1に指示を飛ばす。カフェからでも電車の中からでも、Discordさえ開ければエージェントたちに仕事を振れる。

この構成のポイントは、計算資源の分離。 重い推論はAnthropicのAPIサーバー側で処理され、M1はあくまでオーケストレーションとメッセージングのハブとして機能する。だからメモリ8GBでも問題なく動く。古いマシンでも十分に役割を果たせる。

第四章 エージェントの階層構造

13体のエージェントがただ並んでいるだけでは、組織として機能しない。階層と役割分担が不可欠。

最上位に立つのが「総大将」。 内部名はMyAI。メインエージェントであり、私との会話窓口を担う存在。Discordの指示チャンネルで私のメッセージを受け取り、内容を判断し、適切な処理ルートに振り分ける。

総大将には、OpenClawのheartbeat機能が組み込まれている。これは定期的に状態を確認する仕組み。メールの受信、カレンダーの予定、未処理のタスクがないかを、人間が指示しなくても自律的にチェックする。

次に「第一将」。 Discord上で動く専用ボット。ai議論チャンネルに常駐し、開発タスクの実装を担当する。総大将が「このタスクは開発が必要だ」と判断すると、第一将に引き継がれる。

総大将と第一将をつなぐのが「オーケストレーター」。 orchestrator.jsという名のスクリプトで、両者が議論しながらタスクを進める仕組みを実現している。私が一行の指示を書くだけで、設計の議論から実装、納品まで自律的に進む。

そして、Claw-Empire配下の13体のBoltエージェント。 APIでタスクを投入すると、空いているエージェントが即座にピックアップして処理を開始する。

さらに、特に複雑なタスクに直面した場合、総大将はClaude Code Opus 4.6を自動判断で起動する。 通常タスクにはSonnetやHaikuを使い、本当に難しい問題にだけOpusを投入する。コストと品質のバランスを、エージェント自身が判断して調整する設計。

この階層構造は：

CEO（総大将） → 全体の判断と指示の受付

現場リーダー（第一将） → 開発タスクの実行

実装チーム（13体のBolt） → 並列処理

それぞれの責任範囲が明確で、情報の流れに無駄がない。

第五章 記憶と学習の仕組み

AIエージェントの最大の弱点は、記憶の断絶。セッションが切れれば、それまでの文脈はすべて消える。この問題を、私はファイルベースの記憶システムで解決した。

MEMORY. md → エージェントの長期記憶にあたるファイル。重要な決定事項、学んだ教訓、進行中のプロジェクトの状態が蓄積されている。エージェントはセッション開始時にこのファイルを読み込み、過去の文脈を復元する。

memoryディレクトリ内の日付ファイル（YYYY-MM-DD.md） → 日々の記録が蓄積される。

CLAUDE .mdとlessons .md → エージェントの成長記録。過去の失敗から得た教訓、効率的だった手法が記録されている。使えば使うほど、蓄積された知見によって判断の精度が上がっていく。

この仕組みの重要な点は、「エージェントが自分自身の記憶を編集する」 ということ。heartbeat機能で定期的にMEMORY .mdを見直し、古くなった情報を整理し、新しい知見を追記する。

ある日、総大将が自分のMEMORY .mdに「このプロジェクトではTypeScriptよりPythonのほうが生産性が高かった」と書き込んでいるのを見つけた。私が教えたわけではない。複数のタスクを処理する中で、エージェント自身が導き出した結論。それが記憶として永続化され、以降の判断に反映される。

使うほどに賢くなるシステム。

第六章 運用フローの実際

朝、目が覚める。MacBook Pro M4 Maxを開き、Discordを起動する。指示チャンネルを確認すると、総大将からのメッセージがすでに届いている。

「おはようございます。昨夜22時に受信したメールが3通あります。うち1通は請求書、1通はニュースレター、1通はプロジェクトに関する問い合わせです」

heartbeat機能による自律的なチェック結果。私が寝ている間にメールを確認し、重要度を判断し、要約を作成して報告してくれている。

私は指示チャンネルに書き込む。「昨日話していたAPIの認証機能、今日中に実装してほしい」

総大将がこれを受け取る。開発タスクだと判断し、オーケストレーターを起動。ai議論チャンネルで議論が始まる。設計方針が固まり、第一将が実装に入り、数十分後、納品物チャンネルにコードが投稿される。

この間、私がやったことは三行のメッセージを書いただけ。 設計の議論も、実装も、テストの方針決定も、すべてエージェントたちが自律的に進めてくれた。

並列処理が必要な場面では、Claw-Empireの13体が活躍する。たとえば「このリポジトリの全ファイルにリンターを適用して」といったタスク。13体が同時にファイルを分担して処理し、結果を集約する。単体では時間がかかる作業も、並列化によって劇的に短縮される。

MacBook一台とAPIキーだけで、優秀なアシスタントチームを丸ごと運用している状態。

第七章 コストの真実

OpenClaw本体 → オープンソース。無料。ライセンス費用はゼロ。

Cloudflareトンネル → 無料プラン。

サーバー → 自宅の古いMacBook Pro M1。追加コストはゼロ。電気代はM1チップの省電力性能のおかげで、常時起動していても月に数百円。

唯一の実質的なコストは、AnthropicのAPI利用料。 Claude Sonnet、Haiku、そして時折使うOpus。従量課金制で、使った分だけ請求される。

モデルの使い分けでコストを最適化している：

Haiku → 日常的な軽い処理。応答速度が速く、コストも低い

Sonnet → 開発タスクの議論や実装

Opus → 本当に複雑な問題にだけ投入

この使い分けにより、SaaSのAIツールに月額課金するよりも透明性が高い。使わなければ課金されない。使った分は正確に把握できる。

総合すると、固定費はほぼゼロ。変動費はAPIの従量課金のみ。 クラウドサーバーの月額費用もなければ、SaaSの年間契約もない。

大企業がAIエージェントのインフラに数百万、数千万円を投じている中で、個人がほぼ無料で同等の仕組みを手に入れている。この非対称性こそ、オープンソースとクラウドの民主化がもたらした恩恵。

第八章 つまずきと学び

美しい話ばかりではない。構築の過程では、数え切れないほどの失敗があった。

①エージェント同士の無限ループ。 オーケストレーターの設計が甘かった初期、総大将と第一将が延々と議論を続け、結論に至らないまま会話が膨れ上がる事態が何度も発生した。トークンの消費が爆発的に増え、API費用の請求書を見て青ざめた日もある。→ 解決策：議論にターン制限を設けた。

②記憶の肥大化。 MEMORY. mdにあらゆる情報を書き込んでいたら、ファイルが膨大になり、セッション開始時の読み込みだけで大量のトークンを消費するようになった。→ 解決策：定期的な記憶の整理。heartbeat時にMEMORY.mdを見直し、本当に重要な知見だけを残す運用ルールを確立した。

③Discordのレート制限。 13体のエージェントが同時にメッセージを送信すると、Discordのレート制限に引っかかる。→ 解決策：メッセージキューの実装とバッチ処理への移行。

④私自身の過度な期待。 初期の頃は曖昧な指示で複雑なタスクを丸投げしていた。結果は散々。学んだのは、AIエージェントは「優秀だが文脈を読めない新入社員」に近い存在だということ。→ 解決策：明確な指示と、適切な粒度のタスク分解。

これらの失敗はすべてlessons. mdに記録されている。エージェントたちは次回から同じ失敗を繰り返さない。失敗が資産に変わる仕組み。 これこそが、ファイルベースの記憶システムの真価。

第九章 この体験が変えたもの

AIエージェント運用を始めて数ヶ月。私の働き方は根本的に変わった。

まず、「作業」と「判断」が完全に分離した。 コードを書く作業、調査する作業、ドキュメントを整理する作業。これらはすべてエージェントに委ねられる。私に残るのは、「何を作るか」「どの方向に進むか」という判断だけ。

自分で手を動かすのではなく、チームに方向性を示し、成果をレビューする。しかし、相手は人間のチームではなくAIエージェント。採用も教育も福利厚生も不要。24時間稼働。

次に、「思考の拡張」。 一人では考えつかなかったアプローチを、エージェント同士の議論から得ることがある。ai議論チャンネルで総大将と第一将が交わすやり取りを読んでいると、自分が見落としていた視点に気づかされることが少なくない。

そして、「個人の生産性の上限」が引き上げられた。 以前は一日に一つのプロジェクトしか進められなかった。今は、メインのプロジェクトを進めながら、バックグラウンドで三つ、四つのタスクを同時に走らせている。出力量は数倍に増えた。

ただし、ひとつ重要な気づきがある。このシステムは「AIが人間を置き換える」のではなく、「人間の意思決定を増幅する」仕組み。 方向性を示すのは人間。品質を最終判断するのも人間。エージェントは、その間の実行を圧倒的なスピードで埋める存在。

人間なしでは動かない。しかし、人間一人の力を何倍にも拡張する。

第十章 再現可能性について

「面白い話だけど、自分にもできるのか」

この問いに対する答えは、明確にイエス。

必要なものは四つだけ。

①何かしらのPC。 MacBookでなくてもいい。Linuxマシンでも、Windowsでも。常時起動できる環境があればいい。古くて使っていないPCがあるなら、それで十分。

②AnthropicのAPIキー。 従量課金だから、始めるだけなら数百円の残高で試せる。

③Discordのアカウント。 自分だけのサーバーを作り、チャンネルを整備する。無料。

④GitHubにアクセスしてOpenClawをクローンする基本的なスキル。 ターミナルでgit cloneが打てれば問題ない。

構築にかかる時間は、基本的な環境構築だけなら数時間。私のように13体の階層構造を組むなら数週間。ただし、最初の一体を動かすだけでも十分に面白い。

小さく始めて、育てていく。それが最善のアプローチ。 最初から大規模を目指す必要はない。一体の総大将を動かし、Discordで会話し、heartbeat機能で定期チェックを設定する。それだけで「自分専用のAIアシスタントが自宅サーバーで動いている」という体験が手に入る。

その体験が面白ければ、自然と次のステップに進みたくなる。第一将を追加し、オーケストレーターを組み、並列ワーカーを増やす。気がつけば、大規模な運用体制が出来上がっている。

私がそうだったように。

第十一章 これからの展望

現在のシステムは、まだ進化の途中にある。

次に取り組みたいのは、エージェント間の自律的なタスク生成。 現在は私が指示を出すことでタスクが発生するが、将来的にはエージェント自身が「これをやるべきだ」と判断し、自発的にタスクを生成して実行する仕組みを目指している。

たとえば、コードベースの品質メトリクスを定期的に監視し、劣化を検知したら自動でリファクタリングタスクを起動する。依存ライブラリの脆弱性が報告されたら、自動でアップデートとテストを実行する。人間が気づく前に、問題を発見し、解決まで持っていく。

もう一つの方向性は、外部サービスとの接続の拡充。 現在はDiscordが主な接続先だが、Slackやメール、カレンダー、プロジェクト管理ツールとの統合を深めれば、エージェントの活動範囲はさらに広がる。

そして、長期的に最も興味深いのは「エージェント同士が自律的に最適なタスク配分を行う」状態。13体のエージェントがそれぞれの経験を蓄積し、互いの得意分野を学習し、自律的に最適なタスク配分を行うようになったとき、それはもはや「ツール」ではなく「組織」と呼ぶべきものになる。

その未来は、思ったよりも近い。

おわりに

一台の古いMacBook。

一つのオープンソースフレームワーク。

一つのDiscordサーバー。

それだけで、13体のAIエージェントが連携する自前のインフラが手に入った。

この記事を通じて伝えたかったのは、技術の民主化が「可能性」の段階から「実践」の段階に移っているという事実。 AIエージェントの構築は、もはや大企業の専売特許ではない。個人が、自宅で、ほぼ無料で、始められる。

必要なのは、高価なハードウェアでも、深い機械学習の知識でも、潤沢な予算でもない。「やってみよう」という意思と、数時間の時間だけ。
