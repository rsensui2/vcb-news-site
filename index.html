𝗖𝗼𝗻𝘁𝗶𝗻𝘂𝗲（続行） 同じセッションでそのまま次のメッセージを送る。最も自然な選択肢。 𝟮. /𝗿𝗲𝘄𝗶𝗻𝗱（Esc Escダブルタップ） 過去のメッセージまで戻り、そこからやり直す。それ以降のメッセージはコンテキストから削除される。 𝟯. /𝗰𝗹𝗲𝗮𝗿（新規セッション） 自分で重要な情報をまとめ、まっさらなセッションを開始する。 𝟰. /𝗰𝗼𝗺𝗽𝗮𝗰𝘁（圧縮） これまでの会話をモデルに要約させ、その要約の上に新しい会話を続ける。 𝟱. 𝗦𝘂𝗯𝗮𝗴𝗲𝗻𝘁𝘀（サブエージェント） 次の作業チャンクを独自のクリーンなコンテキストを持つエージェントに委任し、結果だけを親セッションに返す。 ■ 新しいセッションを始めるべきタイミング 100万トークンのコンテキストウィンドウのおかげで、フルスタックアプリをゼロから構築するような長いタスクもより安定して実行できるようになりました。 しかし、コンテキストがまだ余っているからといって、同じセッションを続けるべきとは限りません。 Thariq氏の経験則はシンプルです： 🔥 「新しいタスクを始めるなら、新しいセッションも始めるべき」 ただし例外もあります。たとえば、機能を実装した直後にそのドキュメントを書く場合。新しいセッションを始めるとClaude Codeは先ほど実装したファイルをもう一度読み直す必要があり、遅くてコストもかかります。ドキュメント作成のような「高度な知性を要求しない」タスクなら、多少のコンテキスト汚染を受け入れてそのまま続行する方が効率的です。 ■ 𝗿𝗲𝘄𝗶𝗻𝗱 が最も過小評価されている機能 Thariq氏が「良いコンテキスト管理を示す最大の習慣」として挙げたのが、rewindです。 Claude Codeでは、Escキーをダブルタップ（または/rewindコマンド）することで、過去の任意のメッセージまで戻ってやり直せます。その地点より後のメッセージはコンテキストから完全に削除されます。 具体的なシナリオを考えてみましょう： Claudeが5つのファイルを読み、あるアプローチを試みたが失敗した。 ❌ 普通の反応：「それじゃダメだった。代わりにXを試して」 → 失敗したアプローチの全ログがコンテキストに残り続ける ✅ 良いやり方：ファイル読み込み直後までrewindし、学んだことを含めて再指示 → 「アプローチAは使わないで。fooモジュールはそれを公開していない。直接Bで行って」 こうすることで、失敗した試行錯誤のトークンがコンテキストから消え、クリーンな状態で再挑戦できます。 さらに「ここから要約して」機能を使えば、Claudeが自分の学びをまとめて「過去の自分から未来の自分へのメモ」を作成することもできます。 ■ /𝗰𝗼𝗺𝗽𝗮𝗰𝘁 と /𝗰𝗹𝗲𝗮𝗿 の決定的な違い セッションが長くなった時、重量を減らす方法は2つ。似ているようで全く違います。 【/compact（圧縮）】 ・モデルがこれまでの会話を要約し、履歴をその要約に置き換える ・不可逆（lossy）── middleware、制約はX、関連ファイルはAとB、アプローチYは除外済み」のように ・手間はかかるが、結果のコンテキストは自分が選んだ情報だけで構成される どちらを選ぶかは、コントロールと手軽さのトレードオフです。 ■ なぜ𝗰𝗼𝗺𝗽𝗮𝗰𝘁が失敗するのか 長時間のセッションを使っていると、compactの品質が極端に悪くなることがあります。 ・「サブエージェントでgitの差分からこの機能のドキュメントを書いて」 中間出力を親コンテキストに持ち込まないことで、コンテキストをクリーンに保てます。 ■ まとめ：5つのコンテキスト管理術 Claudeが1ターン終了し、次のメッセージを送る前に必ず判断すべきこと： 1. タスクが変わったか？ → /clear で新規セッション 2. アプローチが失敗したか？ → /rewind 最近、Claude Code ユーザーと話すなかで何度も出てくるテーマがある。1M トークンのコンテキストウィンドウは「諸刃の剣」だという話だ。 これがあるおかげで Claude Code はより長く自律的に動き、タスクを安定して処理できるようになった。しかし同時に、セッション管理を意識しないと「コンテキスト汚染 (context pollution)」への扉も開いてしまう。 セッション管理はこれまで以上に重要になっているが、疑問の声も多い。ターミナルで 1 セッションだけ開くのか、2 つ開くのか？ プロンプトごとに新しく始めるべきか？ compact、rewind、サブエージェントはどう使い分けるのか？ 「悪い compact (bad compact)」はなぜ起きるのか？ この領域には意外と細かい知見があり、Claude Code の使い勝手を大きく左右する。そしてそのほぼすべては、コンテキストウィンドウの管理に起因している。 コンテキスト・コンパクション・コンテキストロット (Context Rot) の簡単な前提整理 コンテキストウィンドウとは、モデルが次の返答を生成するときに「一度に見ている」ものすべてだ。システムプロンプト、ここまでの会話、すべてのツール呼び出しとその出力、読み込まれたすべてのファイルが含まれる。Claude Code のコンテキストウィンドウは 100 万トークンである。 残念ながら、コンテキストを使うことには小さなコストがある。これはしばしば「context rot（コンテキストの劣化）」と呼ばれる。context rot とは、コンテキストが増えるほどモデルのパフォーマンスが落ちていく現象のことだ。注意がより多くのトークンに分散してしまい、古い、いまのタスクと無関係な内容が、いまやろうとしていることからモデルの気を逸らしてしまう。1M コンテキストのモデルでは、おおよそ 30〜40 万トークン付近から多少の context rot が見え始めるが、これはタスク依存であって、厳密な線ではない。 コンテキストウィンドウはハードカットオフなので、末端に近づいたら、ここまでのタスクを小さな要約にまとめて、新しいコンテキストウィンドウで作業を続ける必要がある。これを「コンパクション (compaction)」と呼ぶ。コンパクションは自分で明示的にトリガーすることもできる。 毎ターンが分岐点 (Branching Point) である Claude に何かを頼んで、その作業が完了したとする。いまあなたのコンテキストには、ツール呼び出し・ツール出力・あなたの指示などの情報が溜まっている。ここから次に何をするかの選択肢は、意外と多い。 - Continue — 同じセッションに次のメッセージを送る - /rewind (esc esc) — 過去のメッセージに戻り、そこからやり直す - /clear — 新しいセッションを始める。通常は、いま学んだことを凝縮したブリーフを添えて再スタートする - Compact — ここまでのセッションを要約し、その要約の上に乗って作業を続ける - Subagent — 次の作業の塊を、クリーンなコンテキストを持つ別エージェントに委譲し、結果だけ親に戻す もっとも自然なのは Continue だが、他の 4 つはあなたがコンテキストを管理するために存在している。 新しいセッションを始めるタイミング 新しい 1M コンテキストは、長いタスクをより安定して回せるようになったことを意味する。たとえば、フルスタックアプリをゼロから一気に作らせる、など。しかし、コンテキストが余っているからといって、新しいセッションを始めなくてよい、というわけではない。 私たちの一般的な経験則は、「新しいタスクを始めるときは、新しいセッションも始めるべき」というものだ。 グレーゾーンは、関連するタスクで、一部のコンテキストは必要だが全部は要らない、というケースだ。 たとえば、いま実装したばかりの機能のドキュメントを書くような場合。ここで新しいセッションを始めてもよいが、Claude はいま実装したファイルを再度読み直すことになり、遅くて高価になる。ドキュメント執筆は、必ずしも高度な知性を要するタスクではないので、関連ファイルを読み直さずに済む効率メリットの方が、追加のコンテキストに耐える価値があることが多い。 修正するのではなく「巻き戻し (rewind)」する 良いコンテキスト管理を示す習慣を 1 つだけ挙げろと言われたら、それは rewind だ。 Claude Code では、Esc を 2 回叩く（あるいは /rewind を実行する）ことで、任意の過去のメッセージに戻ってそこから再プロンプトできる。その地点以降のメッセージはコンテキストから削除される。 rewind は、修正アプローチとしてしばしばより優れている。たとえば、Claude が 5 つのファイルを読み、あるアプローチを試したがうまくいかなかった場面を考えてみよう。直感的には「うまくいかなかった。代わりに X を試して」と書きたくなるが、より良い動きはファイル読込直後に rewind し、学んだことを添えて再プロンプトすることだ。「アプローチ A は使うな。foo モジュールはそのメソッドを公開していない。直接 B に行け」。 さらに「ここから要約して (summarize from here)」を使えば、Claude に学習内容を要約させてハンドオフ用のメッセージを作らせることができる。いわば、何かを試してダメだった「過去の Claude」から「これからの Claude」への引き継ぎメモのようなものだ。 コンパクト vs 新規セッション セッションが長くなってきたら、軽量化のルートは 2 つある。/compact か /clear（そしてゼロからやり直し）だ。似ているように見えて、振る舞いはまったく違う。 Compact は、ここまでの会話を要約するようモデルに依頼し、履歴をその要約に置き換える。ロッシーな操作だ。何が重要だったかの判断を Claude に委ねることになるが、自分で要約を書く必要はなく、Claude の方が重要な学びやファイルを網羅的に拾えることもある。指示を添えて方向性を与えることもできる（例: /compact focus on the auth refactor, drop the test debugging）。 /clear では、自分で「何が重要か」を書き出す（「いま auth ミドルウェアをリファクタリングしている。制約は X。重要ファイルは A と B。アプローチ Y は却下済み」）。そしてクリーンな状態から始める。手間はかかるが、残るコンテキストは「自分が関連性ありと判断したもの」だけになる。 「悪い compact」はなぜ起こるのか？ 長時間のセッションを大量に回している人なら、compact が特に悪く働くことがある、と気づいているかもしれない。私たちが観察した限りでは、悪い compact は「モデルがあなたの作業の進む方向を予測できないとき」に起こりやすい。 たとえば、長いデバッグセッションのあとに autocompact がトリガーされ、調査内容が要約されたとする。そこであなたが「じゃあ、さっき bar.ts で見えたあのもう一つの警告も直して」と送ると、元のセッションはデバッグに集中していたため、その警告は要約から落ちている可能性がある。 ここが特にやっかいなのは、context rot の影響で、モデルは compact 実行時にもっとも知性が低い状態にある、という点だ。1M コンテキストでは、あなたが「これから何をしたいか」の説明を添えて /compact を能動的に打つ余裕が増えている。 サブエージェントとクリーンなコンテキストウィンドウ サブエージェントもまた、コンテキスト管理の一形態だ。「この作業の塊では、二度と使わない中間出力がたくさん出る」と事前にわかっているときに便利である。 Claude が Agent ツール経由でサブエージェントを立ち上げると、そのサブエージェントは自前のクリーンなコンテキストウィンドウを持つ。必要なだけ作業して、結果を統合し、最終レポートだけを親に戻すことができる。 私たちが使う判断テスト: 「このツール出力は再び必要か？ それとも結論だけあればよいか？」 Claude Code はサブエージェントを自動で呼び出すこともあるが、明示的に指示したい場面もある。たとえば次のような依頼だ: 「以下の仕様ファイルに基づいて、この作業の結果を検証するサブエージェントを立ち上げて」 「この別のコードベースを読み解き、auth フローがどう実装されているかを要約するサブエージェントを起こして。そのあと、同じやり方で自分でも実装して」 「私の git の変更内容をもとに、この機能のドキュメントを書くサブエージェントを起こして」 まとめ 要するに、Claude が 1 ターンを終えてあなたが新しいメッセージを送ろうとしている瞬間は、常に意思決定ポイントだということだ。 いずれは Claude 自身がこの判断を支援してくれるようになると期待しているが、いまのところ、これが Claude の出力を導くための主要な手段の 1 つである。
