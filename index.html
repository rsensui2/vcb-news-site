タイトル: Claude Codeの検索はなぜ正確なのか？ ベクトル検索を捨てた本当の理由 Cursorなど他のAI開発ツールが採用する「ベクトル検索」を、Anthropicはあえて使わなかった。その理由を、ベクトル検索の仕組みから1つずつ図解で解説する初心者向けガイドです。 1. はじめに：Claude Codeは「ベクトル検索」を使っていない Claude Codeは、コードベースを事前にインデックス化していない。ベクトル埋め込みも使っていない。Cursor も GitHub Copilot（Enterprise）も大半のAI開発ツールはベクトル検索を使っているのに、Anthropicはそのアプローチを捨てた。Boris Cherny（Anthropic, Claude Code 責任者）は「初期のClaude CodeはRAG＋ローカルのベクトルDBを使っていた。だがすぐにエージェント型検索のほうが明らかに優れていることがわかった。シンプルで、セキュリティ・プライバシー・鮮度・信頼性の問題もない」と発言している。 2-3. ベクトル検索の仕組み 言葉や文章を"数字の配列（ベクトル）"に変換し（Embedding）、意味が近いものを数学的に探し出す技術。「犬」「猫」「ウサギ」は同エリア、「車」「飛行機」は別エリアに集まる。質問文もベクトル化され、近傍の文書を取得する（キーワード非一致でも意味類似で検索可能）。 4. 従来型RAGの5ステップと3つの弱点 RAG（Retrieval-Augmented Generation）は高速・意味検索が強みだが、コードの世界では致命的な弱点：①コードは毎日変わるためインデックスがズレる、②コード検索は完全一致が重要（意味類似はノイズ）、③権限管理・暗号化・監査ログで運用負荷が膨らむ。 5. Claude Code の Agentic Search Anthropic公式エンジニアリングブログが「Just-in-Time retrieval」と呼ぶアプローチ。事前にあらゆるデータを処理せず、軽量な識別子（ファイルパス、クエリ、URL）を保持し、ツールで必要なデータだけ実行時に動的ロード。動きは人間のエンジニアと同じ：Glob で候補ファイル列挙 → Grep で全体検索 → Read で有望ファイル読み込み → 外したら別キーワードで再検索。ベクトルDB・埋め込み・事前インデックスは一切不要。 6. 対比 従来型RAG＝「事前に作った地図を見て探す」/ Agentic Search＝「現場に探偵を派遣して調べさせる」。コードベースは毎日再開発される街なので探偵方式が勝つ。 7. 精度が高い5つの理由 ①完全一致が効く（createUserとregisterNewAccountを混同しない）、②常に最新（インデックス陳腐化ゼロ）、③人間のエンジニアの動きと同じ、④モデルが賢くなれば検索も自動で賢くなる（RAGのようなチャンク戦略・埋め込み・リランカーの継続投資不要）、⑤Context Rot（長コンテキストでの注意力劣化）を避け「最小の高シグナルなトークン集合」を組み立てられる。 8. 他のAIツールとの位置づけ キーワード検索 / ベクトル検索 / Agentic Search の比較で、Agentic Search は「外しても再試行できる唯一の方式」。サブエージェントで別視点探索も可能。 9. トレードオフ ①初回レイテンシとトークンコスト（数十万行で5万トークン消費もあり）、②抽象的セマンティック検索はベクトルが上手い（AnthropicはAI推論で補う）、③超大規模コードベースでは苦戦しハイブリッド推奨。 10. まとめ 「事前にまとめるな。必要なときに、必要なだけ取りに行け。」Just-in-Time retrieval が AIエージェント時代の検索設計の軸。社内ドキュメント検索・ナレッジベース・法務・医療など「正確さと鮮度」が効く領域に応用可能。YouTube推薦やECの商品検索のように「似ているものを広く拾う」領域は引き続きベクトルが強い。用途ごとに検索アーキテクチャを選び分ける時代に入った。 参考資料（一次ソース） - Anthropic Engineering Blog「Effective context engineering for AI agents」 - Anthropic Engineering Blog「Building agents with the Claude Agent SDK」 - Claude Code 公式ドキュメント - Boris Cherny本人のX投稿 - Pragmatic Engineer「Building Claude Code with Boris Cherny」
