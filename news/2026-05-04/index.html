Apple — Apple Support アプリ v5.13 に CLAUDE.md を残したまま出荷、緊急修正

Apple Support アプリ v5.13 のバンドル内に、Claude Code が参照する CLAUDE.md（システムプロンプト/ガイドライン定義）が誤って残されたまま配信された。発見後、Apple は数時間以内に当該ファイルを削除した hotfix を緊急配信。Apple が社内開発で Anthropic Claude Code および Mythos preview を実運用している決定的な証拠と受け止められている。

キーポイント

  • 残存していたのは典型的な CLAUDE.md（指示文・システムプロンプト形式）
  • 発見は @aaronp613、複数アカウントが連鎖拡散
  • Apple は同日中に v5.13.x で hotfix 配信し当該ファイルを削除
  • Apple は Anthropic Mythos partner list に既出という観測 (somi_ai)
  • ビルドパイプラインがエージェント生成物（中間ファイル）をフィルタしていなかった点が運用上の教訓
保守的な Apple ですら社内コード作業に Claude Code を使っているという強烈なシグナルとなり、エンタープライズの AI コーディング採用を後押しする。一方で、エージェントが生成する .md / .cursor/ / .claude/ 系ファイルが製品バンドルに混入する事故は他社でも起こり得るため、CI/CD の build artifact フィルタの再点検が必要。

OpenAI — Sam Altman 「Agents SDK 2.0 is underrated」と CEO 自ら再アピール

Sam Altman が短いポストで「Agents SDK 2.0 is underrated」と発信。OpenAI が直近リリースした Agents SDK 2.0 がコミュニティで十分に評価されていないという認識を CEO 自身が表明し、注目を改めて促した。Agents SDK 2.0 は GPT-5.5 / Codex / Workspace Agents を統合するエージェント開発基盤で、ツール定義・サブエージェント・長期記憶・トレースを 1 SDK で扱える。

キーポイント

  • 名指しは Agents SDK 2.0（旧 Agents SDK 1.x の後継）
  • CEO 自身が「過小評価されている」と明示するのは異例
  • 直近の比較トレンドが「Codex CLI vs Claude Code」だったため、SDK レイヤーの議論が薄かった
  • Agents SDK 2.0 は Workspace Agents の裏側でも使われている
  • 発信は 2,003 likes と CEO ポストとしては平均水準だが、開発者層に直接刺さる内容
OpenAI の本気度が「単発のモデル/CLI」ではなくエージェント基盤 (SDK) に向いていることを CEO が再アピール。Vibe Coder Bootcamp 視点では、@openai/agents ベースの実装パターン（ツールオーケストレーション、ハンドオフ、永続メモリ）を抑えることが今後の差別化ポイントになる。

OpenAI 社内 — Codex Auto-review が既定化、Greg Brockman 「コードの 80% は AI が書く」

OpenAI 社内では Codex の Auto-review モード（Codex がコードを自動レビュー → 人間承認は最後だけ）が既定の開発ワークフローになっており、人間承認の回数が約 200 倍削減されたと報告。Greg Brockman は別の発信で OpenAI 社内コード作業のうち AI が書く比率が以前の 20% から 80% に伸びたと公表。AI がコードを書き、AI がレビューし、人間は確認だけ、というループが本格稼働している。

キーポイント

  • Auto-review モードは 2026-04-23 公開、5 月時点で社内既定化
  • Greg Brockman の発言: AI は社内コードの 20% → 80% を書くようになった
  • レビュー承認回数は約 200 倍削減（同じ品質基準で）
  • 「AI が書き、AI がレビューし、人間は Enter を押すだけ」の運用
  • OpenAI が自社で先に実証してから外部展開するパターン
「自動レビューを既定にする」運用が、エンタープライズで AI コーディングを安全にスケールさせる次の標準パターンになり得る。Bootcamp 受講生にとっては、Codex / Claude Code の review-loop（生成 → 自動 review → 修正）を自分のフローに組み込めるかが、生産性差を生む決定的なスキル。

DeepSeek — V4 Pro が Claude Code 経由のリアルコーディングで Opus 4.7 に 6/10 勝利

@teortaxesTex が DeepSeek V4 Pro vs Claude Opus 4.7 を Claude Code 経由で実タスク比較したレポートを公開。実装〜エラー修正までを通しで回した結果、DeepSeek V4 Pro が Opus 4.7 に 6/10 のケースで勝った。Claude 側は「エラーが Claude が読めない形だと自己回復できない」場面で詰まる傾向、という具体観察も付記。

キーポイント

  • 比較対象: DeepSeek V4 Pro vs Claude Opus 4.7（同じ Claude Code クライアント上）
  • 結果: DeepSeek V4 Pro 6 勝 / Opus 4.7 4 勝
  • Opus 4.7 の弱点: 読み取れないエラー形式に詰まり、回復ループに入れない
  • DeepSeek V4 Pro は 4/29 に 75% OFF、Ollama Cloud からも呼べる状態
  • OSS / 中国系フラッグシップが米系プロプライエタリ上位に肉薄〜局所的に逆転する事例
実タスクで OSS が上位プロプライエタリに勝つケースが個人ベンチではなく、Claude Code という確立した実行環境上で起こり始めている。コスト感度の高い受講生にとって、Claude Code から DeepSeek V4 Pro 経由の運用が品質を落とさず費用を 1/4 以下にできるルートとして現実味が増す。
